# AI Verification Through Multi-Representation Cross-Checking

*A design document developed alongside the Dustbuster project.*
*Dustbuster is used here as a low-stakes test environment: an auxiliary Gmail automation system where errors are recoverable and the scope is well-defined.*

---

## The problem this addresses

When an AI system produces output — code, documentation, diagrams, narratives — there is currently no standard lightweight method for checking whether those outputs are mutually consistent, or whether they accurately represent what the system actually does.

Standard verification asks: *does the output meet the requirement?*

This document proposes a complementary question: *do the multiple representations of the same system agree with each other?*

Disagreement between representations is a signal. It may indicate a flaw in the system, a gap in the explanation, or a blind spot in the AI that generated both. Each representation catches different kinds of errors:

- **Narrative** catches logical gaps — things that sound wrong or incomplete when spoken aloud
- **Flowchart** catches structural gaps — missing branches, orphaned decision nodes, unresolved paths
- **Code** catches implementation gaps — what actually runs versus what was described

When all three are generated by the same AI from the same system, cross-checking them is a form of assisted self-verification — with the human as the final arbiter.

---

## Why self-verification has a ceiling

AI systems share a limitation with human experts: self-verification is bounded by what the verifier already knows and believes. An AI that generates incorrect code is unlikely to catch that error by re-reading its own narrative description, because both outputs reflect the same underlying model of the system.

External input — a different representation, a different reader, a different AI — is what catches blind spots.

This is not a failure mode unique to AI. It is a structural property of any self-referential verification system. The method described here does not eliminate this ceiling; it raises it by introducing multiple representations and multiple evaluators.

## The method

### Step 1: Generate three representations of the same system

For any AI-built system, produce:

1. **The code** — the actual implementation
2. **The flowchart** — a visual map of logic flow, decision points, triggers, and states (Mermaid syntax recommended for version control compatibility)
3. **The narrative** — a plain-language story of what the system does, written for a listener rather than a reader; accessible without technical background

All three should be generated from the same source understanding. Ideally, generate them in separate sessions or prompts to reduce the risk of one representation simply echoing another.

### Step 2: Cross-check for consistency

Compare the three representations against each other using these questions:

**Code → Narrative:**
- Does the narrative describe every significant function in the code?
- Are timing cycles (triggers, intervals, deadlines) stated accurately?
- Are conditions described correctly — e.g. "over 75 days" not "after a few months"?
- Are manual human steps clearly distinguished from automated ones?

**Code → Flowchart:**
- Does every decision node in the flowchart correspond to an actual conditional in the code?
- Are all exit paths from decision nodes represented?
- Are there functions in the code with no corresponding node?

**Narrative → Flowchart:**
- Does the narrative describe a logical path that can be traced through the flowchart?
- Are there narrative claims that have no flowchart representation?
- Does the flowchart contain branches the narrative never mentions?

### Step 3: Log discrepancies

Record any mismatch found, with a brief note on which representation is likely correct and why. Categories:

| Type | Description |
|---|---|
| **Precision gap** | Narrative or flowchart is vaguer than the code warrants |
| **Missing branch** | A code path has no representation in flowchart or narrative |
| **Phantom claim** | Narrative or flowchart describes something the code does not do |
| **Timing error** | A duration, interval, or threshold is stated incorrectly |
| **Role confusion** | A manual step is described as automated, or vice versa |

### Step 4: Correct and re-verify

Update the representation(s) that contain the discrepancy. Re-check the specific points of correction against all three outputs before closing the verification pass.

## Multi-AI verification (extended method)

Where possible, submit the same packet — code, flowchart, narrative — to multiple AI systems and ask each to:

1. Identify any inconsistencies between the three representations
2. Note anything the code does that the narrative or flowchart does not describe
3. Note anything the narrative or flowchart claims that the code does not implement
4. Flag any logic in the code that appears to be missing, incomplete, or potentially harmful

Compare the responses across AI systems. Points of agreement increase confidence. Points of disagreement are themselves informative — they reveal which aspects of the system are ambiguous or underspecified.

AI systems to consider including in this comparison:
- The AI that built the system (in a fresh session, without prior context)
- The AI that built the system (in the original session, with full context)
- At least one different AI system (e.g. Gemini, ChatGPT, Grok)

The comparison of responses across systems is itself a dataset worth documenting.

## Dustbuster as test case

Dustbuster is a suitable low-stakes environment for developing this method because:

- It operates on auxiliary Gmail accounts where errors are recoverable
- The scope is well-defined: inbox sorting, outreach tracking, purge lifecycle, learning algorithm
- Three representations already exist: the Apps Script codebase, the Mermaid flowchart, and the narrative story
- It will be open-sourced, making community verification possible

### Known discrepancy (logged during development)

**Type:** Precision gap
**Location:** Narrative, learning algorithm section
**Description:** The narrative states the learning algorithm "finds senders who keep appearing." The code is more specific: it scans the `03-Dumpster-Fire` label, counts sender frequency, and writes structured rows to the Suggested-Rules tab including sender address, frequency count, and a status field.
**Resolution:** Narrative to be updated to reflect the specificity of the implementation.

## On the limits of this document

This document was itself generated by an AI. It should be subject to the same cross-checking process it describes. A human reader should verify that the method proposed here is coherent, that the Dustbuster discrepancy logged above is accurately described, and that no claims are made here that contradict the actual codebase.

The process is not a guarantee of correctness. It is a structured way of raising the probability that errors are caught before they cause harm — and a way of making the verification process legible to people who were not present when the system was built.

## Blog post angle (notes)

*This section is informal — notes toward a piece rather than documentation.*

The interesting claim is not "AI makes mistakes" — that is known. The interesting claim is that AI systems can be made more accountable through multi-representation cross-checking, and that this process also makes their outputs more legible to non-technical audiences. The narrative is not just a communication tool; it is a verification surface. The flowchart is not just a diagram; it is a consistency test. When you ask an AI to explain what it built in three different registers, you learn something about what it actually understands — and where its self-knowledge runs out.

The Dustbuster experiment — submitting the same packet to multiple AI systems and comparing their verification responses — is a small, reproducible demonstration of this. It does not require a research lab. It requires a Gmail account you don't mind experimenting with, a few AI subscriptions, and the patience to read carefully.
